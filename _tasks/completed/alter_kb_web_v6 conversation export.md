### Cause of the Issue
Because the SQL query inside `view_all_pages` does a `LEFT JOIN` on `youtube_videos`, the returned dataset contains columns like `creator`, `video_id`, `duration`, `view_count`, and `thumbnail_url`.
When the server tried to instantiate the Pydantic model with `HTMLPage(**row)`, **Pydantic v2 threw a `ValueError`** because those extra fields were not declared as fields on the model. The server caught this exception in a `try...except` block, logged a message, and silently skipped appending the video object to the list. This is why the creator counts on the pills showed up correctly (computed prior to Pydantic instantiation) but the list was empty.

### Fix Implemented
1. **Added Fields to `HTMLPage`**: Declared the optional YouTube metadata fields (`creator`, `video_id`, `duration`, `view_count`, `thumbnail_url`) directly in the Pydantic model in `src/kb_web/models.py`.
2. **Added Ingestion Serialization Helper**: Created a `_serialize_page_for_db` helper in `src/kb_web/server.py` that strips these YouTube metadata fields before the page data is serialized and upserted/inserted into the SQLite `fetched_pages` table. This keeps the tables strictly decoupled and prevents SQLite from dynamically adding metadata fields to the standard pages table.
3. **Staged & Committed**: Re-ran the unit tests (all passed), confirmed Ruff formatting is clean, and committed the changes to both the submodule and the monorepo root locally.
